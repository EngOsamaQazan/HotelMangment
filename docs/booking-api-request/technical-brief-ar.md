## 1. نبذة تنفيذية

نحن نمتلك ونشغّل **نظام إدارة ممتلكات (PMS) داخلي** مصمَّم خصيصاً لمجموعتنا الفندقية
(3 عقارات — عقار مسجّل حالياً على Booking، وعقاران قيد الإدراج). ونطلب الوصول إلى
Connectivity APIs لنتمكّن من:

- الاحتفاظ **بمصدر حقيقة واحد (Single Source of Truth)** داخل نظامنا لكل ما يخص
  الغرف والأسعار والتوافر والحجوزات — بدون الاعتماد على Channel Manager طرف ثالث.
- توفير **مزامنة ثنائية الاتجاه فوريّة** بين نظامنا و Booking.com.
- التوسّع إلى 3 عقارات تحت نفس حساب المزوّد.

النظام مُشغَّل حالياً في الإنتاج على `https://hotel.aqssat.co` ويستخدمه فريق الاستقبال
والتدبير المنزلي والمحاسبة يومياً.

## 2. نظرة عامة على النظام

| الطبقة              | التقنية                                                  |
|---------------------|----------------------------------------------------------|
| بيئة التشغيل         | Next.js 16 (App Router) على Node.js                      |
| اللغة                | TypeScript (وضع strict)                                  |
| قاعدة البيانات       | PostgreSQL عبر Prisma ORM 6                              |
| المصادقة            | NextAuth.js (JWT، bcrypt، جاهز لـ 2FA)                    |
| التحديث الفوري      | Socket.IO + محفّزات Postgres LISTEN/NOTIFY               |
| الصلاحيات            | RBAC دقيق عبر registry و `requirePermission()`           |
| النشر               | VPS، Nginx كـ reverse proxy، PM2 لإدارة العمليات          |
| تخزين الأسرار        | تشفير AES-256-GCM لكل بيانات الاعتماد الخارجية            |

بيانات اعتماد Booking الخاصة بنا تُخزَّن حصراً مشفّرة بـ AES-256-GCM عبر مفتاح مشتق
بطريقة HKDF من `BOOKING_ENC_KEY` (راجع `src/lib/booking/encryption.ts`).

## 3. نموذج البيانات — التطابق مع Booking Content API

صمّمنا مخطط Prisma بحيث يتطابق 1:1 مع كائنات Booking:

- `UnitType` — نوع الوحدة على مستوى العقار (غرفة فندقية/شقة/جناح)، مع
  `maxOccupancy` و `sizeSqm` و `bedroomCount` و `livingRoomCount` و `bathroomCount`.
- `UnitTypeBed` — تفاصيل الأَسِرَّة لكل نوع (`BED_TYPE`، العدد، الموقع: غرفة نوم /
  صالة / مجلس عربي).
- `Unit` — الوحدة الفعلية المرتبطة بنوع.
- `Rate` — خطة سعر (يومية/أسبوعية/شهرية) مرتبطة بالنوع.
- `Reservation` — حجز الضيف (تواريخ، مدفوعات، القناة، الحالة).
- `Guest` — ملف الضيف الموحّد.

هذا يضمن تحويلاً مباشراً (بدون ترجمة/تطبيع إضافي) إلى
`Properties` و `Rooms` و `Rate Plans` و `Reservations` في توثيق
`https://developers.booking.com/connectivity/docs/`.

## 4. الـ APIs المطلوبة (حسب الأولوية)

1. **Content API** — دفع أنواع الوحدات والأَسِرَّة والمرافق والصور والأوصاف.
2. **Availability & Rates API (ARI)** — دفع التوافر اليومي والأسعار والقيود
   (MinLOS, CTA, CTD).
3. **Reservations API** — سحب الحجوزات الجديدة/المعدّلة وإشعارات الإلغاء.
4. **Messaging API** — (اختياري) لدمج رسائل الضيوف داخل موديول الشات الداخلي.
5. **Opportunities API** — لتحويل الفرص إلى مهام في لوحة Kanban.

لكل تدفّق سنبني: **idempotent consumers**، **exponential backoff**، و**تسجيل
منظَّم** مع correlation IDs.

## 5. وحدات النظام الجاهزة

- **الحجوزات** — CRUD كامل، Check-in/Check-out، عربون، رصيد.
- **أنواع الوحدات والأَسِرَّة** — المخطط أعلاه مع seed data جاهز.
- **تقارير شهرية** — إيرادات، إشغال، ADR، RevPAR، تحليل القنوات.
- **محاسبة ورواتب** — دفتر أستاذ لكل طرف، دورات رواتب لكل موظف.
- **المهام (Kanban)** — لوحات، أعمدة، بطاقات، تحديث فوري.
- **الشات الداخلي** — رسائل مباشرة وقنوات، مع دعم مرفقات.
- **إشعارات فورية** — جرس إشعارات مع عدّاد غير مقروء عبر Socket.IO.
- **سجل صلاحيات** — أكثر من 40 صلاحية مطبَّقة على كل API route.

## 6. الأمن والامتثال

- TLS في كل مكان (شهادات Let's Encrypt متجدِّدة تلقائياً).
- الأسرار: AES-256-GCM في السكون، مع HMAC tag، ولا يوجد تخزين Plain-text.
- سجل تدقيق كامل لكل عملية حساسة (تعديل حجز، تغيير سعر، تعديل صلاحية).
- جاهزية GDPR: يمكن تصدير/إخفاء بيانات الضيف عند الطلب.
- حدّ معدّل الطلبات على الحافة (`src/lib/rateLimit.ts`).
- جاهزون لتوقيع اتفاقية معالجة بيانات (DPA) مع Booking.
